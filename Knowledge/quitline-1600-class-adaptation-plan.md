# Quitline 1600 Brief Adaptation Plan for Class

## Why This Needs Adaptation
The original Quitline 1600 brief is closer to a real digital service project than a normal website design class exercise. It includes:

- UX/UI redesign
- content strategy
- service integration
- CMS and admin requirements
- analytics dashboards
- SEO and marketing setup
- PDPA and consent management

For class, we should not present this as a full delivery scope. We should present it as a **learning case study** that helps students practice **UX thinking, UI design, mobile-first layout, information architecture, and prototype planning**.

Core framing:

> We are not building the full government-scale platform. We are using this brief as a realistic case to learn how to simplify a complex website problem into a clear user-centered design solution.

---

## Best Classroom Positioning

Use this project as:

- a **case-study brief**
- a **UX/UI redesign exercise**
- a **mobile-first service website prototype**
- a **student portfolio-style concept project**

Do not position it as:

- a full production website
- a complete backend system
- a final PDPA-compliant deployed platform
- a full analytics implementation
- a complete chatbot integration project

---

## Classroom Version of the Project

### Suggested Project Title
**Quitline 1600: Mobile-First Quit Support Website Concept**

### Simplified Project Goal
Redesign the Quitline 1600 website concept so users can quickly understand the service and choose the right action:

- call 1600
- add LINE OA
- talk to QuitBot
- read quitting support content
- complete a basic assessment

### Main Learning Focus
Students should learn how to:

- simplify a complex brief
- identify the main user goals
- improve homepage clarity
- build visual hierarchy around CTA actions
- design mobile-first information flow
- create a usable sitemap and wireframe
- translate service goals into interface decisions

## Suggested Teaching Narrative

Use this sequence in class:

### Step 1: Introduce the Real-World Problem
Explain that this is not just a marketing website. It is a **service access website**. Users may need help quickly, so clarity matters more than decoration.

### Step 2: Identify the Core Actions
Ask students:

- What must users do immediately?
- What should be visible in the first screen?
- Which action is primary: Call, LINE, or QuitBot?

### Step 3: Reduce Complexity
Show students that the brief is too large to solve all at once. Teach them to separate:

- essential user-facing experience
- supporting content
- future system requirements

### Step 4: Design the MVP Experience
Turn the project into a simple prototype with:

- clear hero section
- 3 main CTAs
- trust-building explanation
- quick assessment entry
- knowledge hub preview

### Step 5: Reflect on Real vs Classroom Scope
End by showing what a real production project would add later:

- content governance
- analytics dashboards
- CMS workflow
- compliance systems
- service integration

## Suggested MVP Structure

### Homepage
Must answer:

- What is Quitline 1600?
- Who is it for?
- What should I do next?

Suggested homepage blocks:

1. Hero with short value proposition
2. Primary actions:
   - Call 1600
   - Add LINE
   - Talk to QuitBot
3. “Start quitting today” support section
4. Quick assessment / readiness check
5. Knowledge Hub preview
6. Success story or reassurance block
7. FAQ / trust footer

### Secondary Pages

- **About Service**: what the service is and how it works
- **Quit Support Options**: compare call, LINE, chatbot
- **Knowledge Hub**: organized content categories
- **Assessment Entry**: simple action page, not full system design

## Good Classroom Design Questions

Use these questions to blend the brief into class discussion:

- How do we reduce cognitive overload on the homepage?
- What should a stressed or uncertain user see first?
- How can mobile users reach help in 1-2 taps?
- How do we make public-health content feel supportive instead of bureaucratic?
- What content should stay on the homepage, and what should move deeper?
- How do we design trust and clarity without making the page feel heavy?

## Recommended Scope Statement for Class

You can use this exact wording:

> For this class project, we will treat Quitline 1600 as a UX/UI case study. Our goal is not to build the full technical platform, but to redesign the user-facing website experience with a mobile-first approach. We will focus on information clarity, service access, CTA hierarchy, and prototype-level design decisions.

---

## Recommended Evaluation Criteria

Assess student work using:

1. Clarity of homepage message
2. Strength of CTA hierarchy
3. Mobile-first usability
4. Simplicity of navigation
5. Logic of sitemap and flow
6. Quality of wireframes or prototype
7. Ability to justify design decisions from user needs

---

## Final Recommendation

The safest and strongest approach is:

- use the brief as a realistic case
- cut the scope down to frontend UX/UI
- teach prioritization and simplification
- produce wireframes and mockups, not full system promises

In short:

> Blend the brief into the class by turning it from a full service-platform project into a mobile-first UX/UI redesign case study with a clear prototype scope.
